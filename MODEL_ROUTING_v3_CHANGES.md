## 新增内容

### 1. Codex 调用指南

#### 调用时机检查清单

实现新功能前，GLM-5.1 应该自问：

- [ ] 这个功能是否超过 50 行代码？
- [ ] 这个功能是否包含复杂算法？
- [ ] 这个功能是否是核心业务逻辑？
- [ ] 这个功能是否需要深度架构设计？
- [ ] 这个功能是否是 Gate 实现？

**如果任何一项为"是"，则应该调用 Codex 实现。**

#### Codex 实现指令模板

**模板1：实现新 Module**
```
请实现 Phase 1 Module X: [模块名称]

## 背景
[项目背景和目标]

## 技术要求
1. 继承/实现自 [现有类]
2. 方法签名应符合 [接口规范]
3. 错误处理遵循 [项目规范]

## 核心功能
1. 功能A：[详细描述]
2. 功能B：[详细描述]

## 输出要求
- 完整的 Python 代码
- 符合项目编码规范
- 包含必要的 docstring
```

**模板2：实现 Gate**
```
请实现 P1-GX: [Gate名称]

## Gate 目的
[Gate的业务目标]

## 检查项
1. 检查项A：[详细说明]
2. 检查项B：[详细说明]

## 接口要求
- 必须实现 check() 方法
- 返回统一的 GateResult 格式
- 兼容现有的 GateExecutor
```

### 2. 标准开发流程（更新）

```
用户需求
  ↓
GLM-5.1: 任务分解 + 需求分析
  ↓
GLM-5.1: 判断任务类型
  ├─→ Schema/测试/文档 → GLM-5.1 直接实现
  └─→ 核心业务逻辑 → 调用 Codex 实现
       ↓
  Codex: 代码实现
       ↓
  GLM-5.1: 整合代码 + 编写测试
       ↓
  测试验证
       ↓
  Gate实现 → 用户手动 @Notion AI (Opus 4.6)
       ↓
  下一阶段
```

### 3. GLM-5.1 作为"协调者"的职责

1. **任务分解**：将大任务拆解为可执行的小任务
2. **模型选择**：判断每个子任务应该由哪个模型执行
3. **结果整合**：将 Codex 的实现结果整合到项目中
4. **测试编写**：为核心功能编写测试用例
5. **文档维护**：更新相关文档

## v3.0 核心原则

### 原则1：Codex 优先实现核心功能

任何核心业务逻辑、复杂算法、Gate 实现，都应该由 Codex (GPT-5.4) 实现。

### 原则2：GLM-5.1 专注协调工作

GLM-5.1 专注于任务分解、模型调度、结果整合、测试编写。

### 原则3：Opus 4.6 手动审查关键节点

Gate 实现、Phase 完成、架构变更等关键节点，由用户手动在 Notion AI 中触发 Opus 4.6 审查。

---

## 迁移指南

### 对于现有代码

如果代码已经由 GLM-5.1 实现：
1. 评估代码复杂度和重要性
2. 如果是核心功能，考虑由 Codex 重构
3. 保留 GLM-5.1 实现的 Schema 和测试

### 对于新功能

1. GLM-5.1 先进行任务分解
2. 判断是否需要调用 Codex
3. 如果是核心功能，调用 Codex 实现
4. GLM-5.1 编写测试并整合结果
