## Existing pattern: message builder in paraviewProtocol.ts

All protocol messages follow this shape (e.g., createVolumeRenderingToggle):

```typescript
export function createVolumeRenderingToggle(fieldName: string, enabled: boolean): object {
  return {
    id: "pv-volume-toggle",
    method: "visualization.volume.rendering.toggle",
    params: { fieldName, enabled }
  };
}
```

The screenshot RPC uses the built-in `viewport.image.render` method (no custom protocol needed).

## Existing pattern: WebSocket round-trip in ParaViewViewer.tsx

Protocol messages are sent via `sendProtocolMessage(message)` which calls `ws.send(JSON.stringify(message))`.
Response arrives in `ws.onmessage` with `message.id` matching the request id.

## Screenshot RPC details (ParaView Web built-in)

- Method: `viewport.image.render`
- Params: `{ quality: 95, size: [viewportWidth, viewportHeight] }` — the `size` param is what ensures SHOT-01.2 (captures at viewport resolution, not internal buffer)
- Response: `{ id, result: { image: "base64-encoded PNG data" } }` — the base64 string IS the PNG data
- After receiving: decode base64 -> create Blob -> trigger anchor download -> revoke object URL
- Filename: `cfd-screenshot-{YYYYMMDDHHMMSS}.png`

5. **Handle screenshot response in ws.onmessage** — add inside the existing `ws.onmessage` handler, after the `pv-volume-status` block:
   ```typescript
   // SHOT-01.1: Parse screenshot response and trigger download
   if (message.id === 'pv-screenshot' && message.result) {
     const r = message.result as Record<string, unknown>;
     const base64Data = typeof r.image === 'string' ? r.image : null;
     if (base64Data) {
       // Decode base64 to Blob and trigger download
       const byteString = atob(base64Data);
       const ab = new ArrayBuffer(byteString.length);
       const ia = new Uint8Array(ab);
       for (let i = 0; i < byteString.length; i++) {
         ia[i] = byteString.charCodeAt(i);
       }
       const blob = new Blob([ab], { type: 'image/png' });
       const url = URL.createObjectURL(blob);
       const link = document.createElement('a');
       link.href = url;
       const timestamp = new Date().toISOString().replace(/[-:T]/g, '').slice(0, 14);
       link.download = `cfd-screenshot-${timestamp}.png`;
       document.body.appendChild(link);
       link.click();
       document.body.removeChild(link);
       URL.revokeObjectURL(url);
     }
     // SHOT-01.4: Re-enable after 500ms debounce window
     screenshotTimeoutRef.current = setTimeout(() => {
       setScreenshotCapturing(false);
       screenshotTimeoutRef.current = null;
     }, 500);
   }
   ```

   Note: SHOT-01.4 debounce works as follows — button is disabled immediately on click. After 500ms the button re-enables. If user clicks again within that 500ms window, the `screenshotTimeoutRef.current` guard at the top of `handleScreenshot` catches it (since it is still non-null) and returns early.

|----------|-------------|
| client -> server (viewport.image.render) | Server returns base64 PNG; client decodes and downloads — no server-side file write |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|----------------|
| T-21-01 | Tampering | Screenshot download | accept | PNG is rendered read-only view; no file write to server; client download is isolated to browser Downloads folder |
| T-21-02 | Denial of Service | Large viewport screenshot | mitigate | Screenshot resolution capped at actual `offsetWidth/offsetHeight` (not arbitrary max); debounce limits rapid requests; server-side ParaView Web handles its own resource limits |
| T-21-03 | Information Disclosure | Screenshot contains CFD data | accept | User is intentionally capturing their own data; no external exposure |
