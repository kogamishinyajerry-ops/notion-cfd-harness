## Researcher Decisions

The objective prompt asked the UI researcher to resolve three open questions. Decisions are locked below:

| Question | Decision | Rationale |
|----------|----------|-----------|
| Sweep definition UI: modal wizard or separate page? | **Separate page** — `SweepCreatePage.tsx` | Matches the `PipelineCreatePage.tsx` pattern already established in Phase 31. Separate pages for creation flows is the project convention. |
| Progress view: embedded in PipelineDetailPage or standalone? | **Standalone `SweepDetailPage.tsx`** | A sweep manages N pipeline instances, not one. It needs its own header, aggregate progress bar, per-combination card list, and summary table. Embedding in PipelineDetailPage would break the 1:1 pipeline-detail relationship. |
| Summary table: expandable inline or separate tab? | **"Summary" tab inside SweepDetailPage** | Follows the exact tab pattern in `PipelineDetailPage.tsx` (steps / events / config). Three tabs: Combinations \| Summary \| Config. |

## Color

Inherited from `dashboard/src/theme.css`. No new palette values introduced.

| Role | Light Value | Dark Value | Usage |
|------|-------------|-----------|-------|
| Dominant (60%) | `--bg-primary` (#ffffff) | (#1d1d1f) | Page backgrounds, form backgrounds |
| Secondary (30%) | `--bg-secondary` (#f5f5f7) | (#2a2a2e) | Sweep cards, combination cards, table rows (alternating) |
| Accent (10%) | `--accent-color` (#e94560) | (#e94560) | Reserved for specific elements listed below |
| Destructive | `--color-failed` (#ef4444) | (#ef4444) | Cancel sweep confirmation text + border |

**Accent (#e94560) reserved for:**
- "Run Sweep" primary CTA button (`.btn.btn-primary` background)
- Active filter pill highlight on SweepsPage
- Active tab underline indicator on SweepDetailPage

**Accent is explicitly NOT used for:**
- Status badges (use existing `--color-*` tokens)
- Table sort arrows
- General hover states (use `--accent-hover: #d63d56` only on `.btn-primary:hover`)

**Status color tokens reused from theme.css (no new tokens):**

| Status | Token | Value | Applied To |
|--------|-------|-------|-----------|
| PENDING / QUEUED | `--color-pending` | #9ca3af | Combination awaiting concurrency slot |
| RUNNING | `--color-running` | #f59e0b | Active combination pipeline |
| COMPLETED | `--color-completed` | #10b981 | Finished combination |
| FAILED | `--color-failed` | #ef4444 | Failed combination |
| CANCELLED | `--color-cancelled` | #6b7280 | Cancelled combination |
| PAUSED | `--color-paused` | #8b5cf6 | Sweep-level paused state |

**New semantic token (add to theme.css):**

```css
--color-sweep-active: #3b82f6; /* Sweep is actively dispatching combinations */
```

Used only for the sweep-level status badge when the sweep itself is running but not all combinations have started.

## Component Specifications

### SweepsPage

Layout mirrors `PipelinesPage.tsx` exactly:
- Header row: h1 "Sweeps" + "New Sweep" CTA button (right-aligned)
- Filter bar: pill buttons — All | PENDING | RUNNING | COMPLETED | FAILED
- Sweep card list (`.sweep-card`): same card pattern as `.pipeline-card`
  - Card fields: sweep name, status badge, base pipeline name, combination count, created_at, "View Details" link + "Delete" button
  - Left border color keyed to sweep status (`.border-left-{status}`)

Empty state: centered block, no icon, heading + body + CTA (matches existing PipelinesPage empty state pattern).

### SweepCreatePage

Two form sections, same layout as `PipelineCreatePage.tsx`:

**Section 1 — Sweep Settings:**
- Sweep Name (text input, required, maxLength=64)
- Description (textarea, optional, maxLength=256, rows=2)
- Base Pipeline (select dropdown — populated from `GET /pipelines?status=PENDING` — shows pipeline name + id)
- Max Concurrent Containers (number input, min=1, max=10, default=2, label: "Max concurrent Docker containers")

**Section 2 — Parameter Grid:**
- Dynamic row list. Each row has:
  - Parameter name (text input, placeholder: "e.g. velocity", width: 180px fixed)
  - Values (text input, placeholder: "e.g. 1, 2, 5  — comma-separated", flex: 1)
  - Remove button (text: "Remove", `.btn-remove`)
- "Add Parameter" button below rows (`.btn-add-step` class reuse)
- Combination count preview chip: reads as "**N combinations** will be run" — computed live from `itertools.product` expansion count, shown in a `--bg-tertiary` rounded chip below the parameter rows. Updates on every keystroke.

Form validation errors: rendered in `.form-error-banner` (reuse existing class).

Form actions row: "Cancel" (link to `/sweeps`) + "Run Sweep" primary CTA button.

### SweepDetailPage

Layout mirrors `PipelineDetailPage.tsx`:

**Header block:**
- Back link: `← Back to Sweeps`
- h1: sweep name
- Status badge (sweep-level)
- WS/polling indicator (same pattern as PipelineDetailPage)

**Meta grid** (same 6-cell grid as PipelineDetailPage):
- Sweep ID, Status, Base Pipeline (link to `/pipelines/{id}`), Created, Combinations, Duration

**Control bar:**
- PENDING → "Start Sweep" (`.btn.btn-primary`)
- RUNNING → "Cancel Sweep" (`.btn.btn-outline-danger`)
- COMPLETED / FAILED / CANCELLED → no controls

**Aggregate progress bar:**
- Same `.pipeline-progress` pattern
- Label: "M / N combinations complete (P%)"
- Progress fill uses `--color-completed` (#10b981)
- Visible only when sweep status is RUNNING

**Tab navigation (3 tabs):**

```
[ Combinations ]  [ Summary ]  [ Config ]
```

Same `.job-content-tabs` + `.tab-btn` + `.tab-content` classes.

**Combinations tab:**
- Grid or list of per-combination cards (`.combination-card`)
- Each card shows: combination hash (first 8 chars), param values badge list, status badge, duration
- Status badge color matches `--color-{status}` tokens
- Cards are sorted: RUNNING first, then PENDING, then COMPLETED/FAILED

**Summary tab (shown after sweep starts, full data when COMPLETED):**
- HTML table: columns = Case ID | Params | Final Residual | Status | Duration
- Case ID links to the combination's pipeline detail: `/pipelines/{pipeline_id}`
- Params column: compact key=value pairs, comma-separated (e.g. `velocity=1, resolution=50`)
- Final Residual: formatted to 4 significant digits or "—" if not yet available
- Status: colored badge using existing status tokens
- Table is sortable by clicking column headers (client-side sort, no pagination — sweep combinations ≤ 1000)
- Export button: "Export CSV" — triggers browser download of table as `.csv`

**Config tab:**
- Same key-value list pattern as PipelineDetailPage config tab
- Shows: Sweep ID, Base Pipeline ID, Max Concurrent, Parameter Grid (JSON pre block), Created, Updated

## Interaction States

| State | Behavior |
|-------|----------|
| Sweep PENDING | Start button visible; combinations show as QUEUED badges |
| Sweep RUNNING | Cancel button visible; aggregate progress bar animates; combination cards update via WebSocket polling fallback (10s) |
| Sweep COMPLETED | No control buttons; all combination cards show final status; Summary tab is fully populated |
| Combination QUEUED (waiting for concurrency slot) | Shows `--color-pending` badge with label "QUEUED" |
| Combination RUNNING | Shows `--color-running` badge; no per-combination progress bar (too noisy) |
| Combination FAILED | Card has `border-left-failed` red left border |
| Parameter row value parse error | Inline error text below the Values input: "Enter comma-separated numbers (e.g. 1, 2, 5)" in `--color-failed` |
| Combination count = 0 (no params defined) | Combination preview chip reads: "Define at least one parameter to run a sweep" |
| Combination count > 50 | Preview chip adds warning: "50+ combinations — this sweep may take a long time" |

---

## Real-time Update Strategy

Sweeps do not add a new WebSocket endpoint. The `SweepDetailPage` polls `GET /sweeps/{id}` every 10 seconds for aggregate status (same polling interval as `PipelinesPage`). Individual combination pipelines are not subscribed via WebSocket on the sweep page — their detail is accessible by navigating to `/pipelines/{pipeline_id}` via the Case ID link in the Summary table.

This avoids N simultaneous WebSocket connections for large sweeps.
